# 事务

# 什么是事务

​ 事务只是一个改变，是一些操作的集合。用专业的术语讲，它就是一个程序的执行单元。事务的本身其实并不包含ACID这4个特性,我们需要通过某些手段，尽可能的让这个执行单元去满足这4个特性，那么，我们就可以称他为一个事务，或者说是一个正确的事务，完美的事务。

# 事务的四大特性（ACID）

  1. 原子性（Atomicity）

    原子性是指事务里包含的所有操作，要么全部成功，要么就全部失败回滚。因此事务的操作如果成功就必须完全应用到数据库中，如果操作失败，就不能对数据库有任何影响。

  2. 一致性 （Consistency）

    一致性指的就是一个事务执行之前和之后都必须处于一致的状态。

    就拿转账来说，用户A和用户B的钱想加起来是5000，那么不管A和B怎么转账，或者转几次，事务结束之后两个用户的钱相加起来还得是5000，这就是事务的一致性。

  3. 隔离性 （lsolation）

    隔离性指的是多个并发事务之间是相互隔离，比如多个事务操作同一张表，每一个事务都不能被其他的操作所干扰。MySQL这块还有4种事务的隔离级别

  4. 持久性 （Durability）

    持久性是指当事务一旦被提交了，那么对数据库的改变是永久性的。

# 事务的隔离级别

为了权衡隔离和并发的矛盾，ISO定义4个事务的隔离级别，每个隔离级别的隔离程度各不同，允许出现的副作用也不同。

隔离级别 脏读 不可重复读 幻读
读未提交（Read-uncommitted） 可能 可能 可能
读已提交（Read-committed） 不能 可能 可能
可重复读（Repeatable Read） 不能 不能 可能
可串行化（Seralizable） 不能 不能 不能

  • 读未提交（read-uncommitted）

    最低的一个级别，事务A可以读取到事务B修改但未提交的数据。

    一般很少用这种隔离级别，除非是浏览量文章阅读量每次+1这种场景。

  • 读已提交（read committed）

    事务A只能读取到事务B修改并已提交的数据。

    如果希望在同一个事务中，数据不管读取多少次，结果都是一样的，这种隔离级别就无法满足。

  • 可重复读（repeatable read）（默认的隔离级别）

    在同一个事务中，多次读取数据时，不管其他的事务有没有修改过数据（不论是否提交）始终是一样的值。

  • serializable 串行化

一个事务在修改数据的时候，如果其他的事务也想改，就必须等前边的事务提交或者回滚以后才能继续。

最严格的级别，性能最低，也很少用。

#创建数据表
CREATE TABLE `account` (
		`id` INT ( 6 ) UNSIGNED auto_increment,
		`name` VARCHAR ( 100 ) NOT NULL,
		`money` DECIMAL ( 9, 2 ) NOT NULL,
		PRIMARY KEY ( `id` ),
		UNIQUE KEY `name` ( `name` ) USING BTREE 
	) ENGINE = INNODB DEFAULT CHARSET = utf8;
#插入测试数据

  • 脏读：A事务还没有commit B事务就读到了A事务的数据（破坏了隔离性）

    #例：A用户是买家B是卖家，A要付100元给B的场景

#第一步 先打开两个终端，同时登陆到MySQL上去，第一个终端模拟A用户，第二个终端模拟B用户，此时的两个用户的数据如下：
+----+------+---------+
| id | name | money   |
+----+------+---------+
|  1 | A    | 1000.00 |
|  2 | B    | 1000.00 |
+----+------+---------+

#第二步 在B用户(卖家)的终端设置 隔离级别 为最低级别 读未提交(read uncommitted) 因为只有最低级别才支持脏读，执行SQL：
set session transaction isolation level Read uncommitted; #设置级别为读未提交

#第三步 在A用户终端执行SQL：
start transaction; #开始事务 begin or start transaction,begin是start transaction启动事务的别名，start... 是标准的SQL语法，是启动临时事务的推荐方法。
update account set money=money-100 where name="A";#扣掉A100元
update account set money=money+100 where name="B";#给B加100元	
#第四步 在B用户的终端同样开始事务（start transaction），然后查询两个用户的数据如下：
+----+------+---------+
| id | name | money   |
+----+------+---------+
|  1 | A    |  900.00 |
|  2 | B    | 1100.00 |
+----+------+---------+
#可以发现，A用户的事务还未 commit 事务，B用户已经读取到了A用户uncommitted事务的数据，此读取到的数据，称为脏读（假象，不真实，的一个数据）

  • 不可重复读：A事务在本次事务中，对自己未操作过的数据进行了多次读取，结果出现了不一致，或者不存在的情况。（破坏了一致性，update和delete）

  • 幻读：A事务在本次事务中，对自己未操作过得数据进行了多次读取，第一次读取时，第二次读取时记录出现了。（破坏了一致性，insert）
